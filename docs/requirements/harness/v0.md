# IonDB — Harness Requirements (v0)

The development harness for agent-driven development of IonDB. See `docs/reference/harness-engineering.md` for the conceptual foundation.

---

## 1. Purpose

IonDB is built agent-first. The harness is the system of context, constraints, feedback loops, and governance that ensures agents produce correct, consistent, architecturally coherent code — from bare-metal Cortex-M0 firmware to full-featured Linux builds.

The harness must handle IonDB's defining challenge: a single codebase that spans three hardware tiers (2 KB RAM microcontrollers through 256 MB+ embedded Linux), controlled by a combinatorial feature-flag system, with ACID correctness guarantees that must hold under crash, power loss, and concurrent access.

## 2. Context Engineering

### 2.1 CLAUDE.md as a Map

The root `CLAUDE.md` serves as a table of contents — not an encyclopedia. It must:

- Be under 150 lines.
- Point to `docs/requirements/` for specifications, `docs/reference/` for conceptual guidance, and per-crate `CLAUDE.md` files for crate-specific context.
- Declare the workspace structure, dependency rules, and feature-flag system at a glance.
- List the commands an agent needs to build, test, lint, and verify.

### 2.2 Per-Crate Context Files

Each workspace crate (`iondb-core`, `iondb-alloc`, `iondb-storage`, `iondb-wal`, `iondb-tx`, `iondb-buffer`, `iondb-query`, `iondb-io`, `iondb`) contains a `CLAUDE.md` that describes:

- The crate's role and architectural position (which layer it occupies, what it depends on, what depends on it).
- Key traits defined or implemented in this crate.
- Feature flags that affect this crate.
- Target tier compatibility (`no_std`, `alloc`, `std`).
- Common pitfalls and non-obvious constraints specific to this crate.

### 2.3 Repository as Knowledge Base

All project knowledge lives in the repo as versioned artifacts:

| Location | Content |
|---|---|
| `docs/requirements/initial/` | Core product requirements (functional, non-functional, architecture). |
| `docs/requirements/harness/` | This document — harness requirements. |
| `docs/reference/` | Conceptual reference documents (harness engineering, architecture decisions). |
| `CLAUDE.md` (root) | Top-level agent context map. |
| `*/CLAUDE.md` (per-crate) | Crate-specific agent context. |

If information isn't discoverable to the agent, it effectively doesn't exist.

### 2.4 Context Injection Through Tooling

Documentation alone is insufficient. The harness injects context at the point of failure:

- **Custom Clippy lints** include remediation instructions in their error messages. When an agent violates a constraint, the error tells it exactly how to fix the violation.
- **Test failure messages** explain the expected invariant, not just the assertion that failed. Example: a dependency-direction test failure says "iondb-storage must not depend on iondb-wal — cross-crate integration belongs in iondb-tx or the facade crate" rather than just "unexpected dependency found."
- **Build failure messages** for `no_std` targets explain which `std`-dependent code path was reached and how to gate it behind a feature flag.

## 3. Architectural Constraints

### 3.1 Dependency Layers

IonDB's crate structure defines a strict dependency direction, enforced mechanically:

```
iondb-core          (depends on nothing — leaf of the dependency tree)
    ↑
iondb-alloc         (depends on iondb-core)
iondb-io            (depends on iondb-core)
iondb-storage       (depends on iondb-core)
iondb-wal           (depends on iondb-core)
iondb-buffer        (depends on iondb-core)
iondb-query         (depends on iondb-core)
    ↑
iondb-tx            (depends on iondb-core — cross-cutting integration)
    ↑
iondb               (facade — re-exports, convenience API)
```

**Rules enforced by the harness:**

1. `iondb-core` depends on nothing (except `core`/`alloc`).
2. All implementation crates depend only on `iondb-core` for shared traits and types.
3. No horizontal dependencies between implementation crates (e.g., `iondb-storage` must not depend on `iondb-wal`).
4. Cross-cutting integration happens only in `iondb-tx` and the facade `iondb` crate.
5. `unsafe` code is confined to `iondb-alloc` and `iondb-io`.

These rules are validated by structural tests (see 3.3) — not just documented.

### 3.2 Custom Linters

Custom lint rules with agent-oriented remediation messages:

| Rule | Scope | Remediation Message |
|---|---|---|
| **No `std` import without feature gate** | All crates | "This code path uses `std`. Gate it behind `#[cfg(feature = \"std\")]` or use the `core`/`alloc` equivalent." |
| **No `format!`/`panic!` with formatting in `no_std`** | All crates (when `std` disabled) | "String formatting pulls in `core::fmt` machinery and bloats binary size. Use `defmt` for logging or return a typed error variant." |
| **No `Box<dyn Error>`** | All crates | "Use `iondb_core::Error` enum. Error variants behind feature flags are compiled out when unused." |
| **No unaligned reads/writes** | `iondb-io`, `iondb-alloc` | "All buffer accesses must go through safe accessors. Use `read_unaligned`/`write_unaligned` or align the buffer." |
| **No `#[allow]` without justification** | All crates | "Suppressing a Clippy lint requires a justifying comment approved in code review. Remove the `#[allow]` or add a comment explaining why it's necessary." |
| **No cross-crate horizontal dependency** | Implementation crates | "This crate depends on another implementation crate. Cross-crate integration belongs in `iondb-tx` or the facade `iondb` crate. See `docs/requirements/initial/v0.md` §7.2." |
| **File size limit** | All crates | "This file exceeds 500 lines. Split it into smaller modules with clear responsibilities." |

### 3.3 Structural Tests

Tests that validate the shape of the codebase, run as part of `cargo test`:

| Test | What It Validates |
|---|---|
| **Dependency direction** | Parses `Cargo.toml` files and asserts that no implementation crate depends on another implementation crate. |
| **`no_std` compilation** | `cargo build --no-default-features --target thumbv6m-none-eabi` succeeds for `iondb-core`, `iondb-alloc`, `iondb-storage`, `iondb-wal`, `iondb-buffer`. |
| **Feature-flag isolation** | Each feature flag can be enabled independently without pulling in unintended dependencies. |
| **Public API surface** | All public types, traits, and functions have documentation. `RUSTDOCFLAGS="-D warnings" cargo doc --no-deps` passes. |
| **Zero-allocation verification** | Tests using a counting `#[global_allocator]` assert that `profile-minimal` code paths perform zero heap allocations. |
| **Binary size budget** | `profile-minimal` build for `thumbv6m-none-eabi` produces < 32 KB `.text` section. |
| **Memory budget** | Core engine operational with <= 2 KB RAM (excluding user data pages) for Tier 1. |
| **Naming conventions** | Module names, trait names, and error variants follow project conventions. |

### 3.4 Taste Invariants

Non-negotiable design rules enforced mechanically:

| Invariant | Enforcement |
|---|---|
| **`no_std` by default** | `thumbv6m-none-eabi` build in CI. If it doesn't compile for Cortex-M0, it's broken. |
| **Pay only for what you use** | Feature flags must gate all optional functionality. Unused engines/allocators/backends add zero bytes. |
| **Pluggable everything** | All extension points go through traits (`StorageEngine`, `MemoryAllocator`, `IoBackend`, `Codec`). No concrete type coupling across crate boundaries. |
| **Correctness first** | ACID guarantees backed by tests. WAL write-ahead invariant: no dirty page flushed before its log record is synced. |
| **Zero compiler warnings** | `RUSTFLAGS="-D warnings"` on all builds. Zero Clippy warnings. Zero doc warnings. |
| **100% unit test coverage** | Measured via `cargo-tarpaulin` or `llvm-cov`. Any drop blocks the merge. |
| **Explicit error handling** | Single `iondb_core::Error` enum. No `unwrap()` or `expect()` in library code. |
| **Endianness-aware formats** | All on-disk formats explicitly define byte order (little-endian default). |
| **Alignment-safe access** | No unaligned reads/writes. All buffer accesses through safe accessors. |

## 4. Feedback Loops

### 4.1 Inner Loop — Agent Session

The tightest feedback cycle. The agent writes code, runs checks, reads errors, fixes issues, and iterates autonomously.

**Quality gates available in the inner loop:**

| Gate | Command | Feedback |
|---|---|---|
| **Compile (host)** | `cargo build` | Type errors, borrow checker errors, feature-flag conflicts. |
| **Compile (`no_std`)** | `cargo build --no-default-features --target thumbv6m-none-eabi` | `std` leakage, missing `no_std` compat. |
| **Unit tests** | `cargo test` | Functional correctness on host. |
| **Clippy** | `cargo clippy --all-targets --all-features -- -D warnings` | Style, correctness, performance lints. |
| **Formatting** | `cargo fmt --all -- --check` | Formatting compliance. |
| **Doc check** | `RUSTDOCFLAGS="-D warnings" cargo doc --no-deps` | Missing or broken documentation. |
| **QEMU tests** | `cargo test --target thumbv6m-none-eabi` | Tier 1 functional correctness on emulated hardware. |
| **Miri** | `cargo +nightly miri test -p iondb-alloc` | Undefined behavior in `unsafe` code. |

Each gate produces error messages with remediation instructions. The agent self-corrects without human intervention.

### 4.2 Middle Loop — PR / Review

A PR is opened. CI runs the full verification stack:

| CI Job | What It Runs |
|---|---|
| **Host-native** | `cargo test`, `cargo clippy`, `cargo fmt --check`, `cargo doc` |
| **Feature matrix** | Build and test every meaningful feature-flag combination |
| **QEMU ARM** | `cargo test --target thumbv6m-none-eabi`, `cargo test --target thumbv7em-none-eabihf` |
| **QEMU RISC-V** | `cargo test --target riscv32imac-unknown-none-elf` |
| **Renode** | Flash I/O round-trip, wear-leveling verification |
| **Miri** | UB detection in allocators and `unsafe` code |
| **Loom** | Thread interleaving tests for `concurrency`-flagged code |
| **Coverage** | 100% line coverage — any drop blocks merge |
| **Binary size** | `profile-minimal` < 32 KB `.text` on `thumbv6m-none-eabi` |
| **Structural tests** | Dependency direction, naming conventions, architectural compliance |
| **Dogfood apps** | Build and test `sensor-log`, `edge-config`, `fleet-telemetry` |

Reports are generated. Review happens. Issues are fixed and re-validated.

### 4.3 Outer Loop — Harness Improvement

The meta-loop. Humans observe patterns in agent failures and improve the harness:

- Agent repeatedly introduces `std` dependency in `no_std` code → add a custom lint with remediation message.
- Agent misunderstands transaction lifecycle → improve `iondb-tx/CLAUDE.md` with typestate pattern explanation.
- Agent creates oversized modules → add file-size structural test.
- Agent forgets to gate code behind feature flags → add feature-flag isolation test.

Every agent failure is a harness improvement opportunity. The harness is never "done."

## 5. Garbage Collection

### 5.1 Recurring Sweep Agents

Automated background tasks that fight entropy:

| Sweep | Cadence | What It Does |
|---|---|---|
| **Dead code** | Per PR | Detect unused functions, types, imports, and feature-flag-gated code that is never enabled. |
| **Documentation staleness** | Weekly | Compare doc comments and `CLAUDE.md` files against actual code. Flag drift. |
| **Test gap analysis** | Per PR | Identify new public APIs without corresponding tests. |
| **Dependency audit** | Weekly | Check for new transitive dependencies that pull in `std` when `std` is disabled. |
| **Feature-flag hygiene** | Weekly | Verify that every feature flag is tested in the CI matrix and documented in `docs/requirements/initial/v0.md` §5. |
| **Pattern violation scan** | Per PR | Check for `unwrap()` in library code, `Box<dyn Error>`, unaligned access patterns, missing error variants. |
| **Binary size regression** | Per PR | Compare `.text` section size against baseline. Flag regressions > 5%. |

### 5.2 Continuous Debt Payment

Technical debt is addressed incrementally:

- Sweep agents open small, targeted cleanup PRs — most reviewable in under a minute.
- No "cleanup Fridays." Continuous small increments beat painful periodic bursts.
- Each cleanup PR links to the pattern it addresses and the harness improvement that prevents recurrence.

## 6. IonDB-Specific Harness Extensions

### 6.1 Cross-Tier Compilation as Constraint

The `thumbv6m-none-eabi` build is the ultimate architectural constraint. If code compiles for Cortex-M0 with `no_std` and no heap, it respects every memory and dependency rule by construction. This single build target enforces:

- No `std` dependency leakage.
- No heap allocation without `alloc` feature.
- No `format!`/`panic!` with formatting.
- No unaligned access (Cortex-M0 faults on unaligned access).
- Binary size within budget.

### 6.2 Feature-Flag Combinatorics

IonDB's feature flags create a combinatorial space of valid configurations. The harness must verify meaningful combinations, not the full power set:

| Profile | Feature Flags | Target |
|---|---|---|
| `profile-minimal` | `alloc-static`, `storage-bptree`, `io-mem` | `thumbv6m-none-eabi` |
| `profile-embedded` | `alloc`, `alloc-bump`, `storage-bptree`, `storage-hash-ext`, `storage-hash-linear`, `io-raw`, `wal`, `buffer-pool`, `query` | `thumbv7em-none-eabihf` |
| `profile-full` | Everything | Host-native |
| **Individual flags** | Each flag enabled alone | Host-native |
| **Incompatible pairs** | `alloc-system` without `std`, `io-file` without `std` | Must fail to compile with clear error |

A CI script generates the feature matrix automatically — it is not manually maintained.

### 6.3 Resource Budget Enforcement

| Budget | Target | Enforcement |
|---|---|---|
| **RAM** | <= 2 KB baseline for Tier 1 | CI script parses linker map file, asserts `.bss` + `.data` <= 2048 bytes. |
| **Binary size** | < 32 KB `.text` for `profile-minimal` | CI script parses `size` output, asserts `.text` < 32768 bytes. |
| **Zero allocation** | `profile-minimal` code paths | Counting `#[global_allocator]` in tests asserts zero heap allocations. |
| **Static analysis** | `profile-minimal` build | `cargo build` reports zero dynamic allocations. |

### 6.4 Crash Simulation

The `FailpointIoBackend` wraps any `IoBackend` and injects faults at configurable points:

- I/O error before write.
- I/O error after write but before sync.
- Partial write (truncated).
- Sync failure.
- Read corruption (bit flip).

Crash simulation tests exercise every recovery path:

- WAL replay after crash mid-write.
- Partial checkpoint detection and recovery.
- Corrupted storage file detection via checksums.
- Incomplete transaction rollback.

These tests produce deterministic, reproducible results. The agent can run them, read failures, and iterate.

### 6.5 Simulator Environments

| Environment | Tier | Purpose |
|---|---|---|
| **QEMU** (`lm3s6965evb`) | Tier 1 | Cortex-M0 bare-metal test execution |
| **QEMU** (`netduinoplus2`) | Tier 2 | Cortex-M4F test execution with FPU |
| **QEMU** (`virt` RISC-V) | Tier 1 | RISC-V `no_std` validation |
| **Renode** | Tier 1/2 | Full SoC simulation with flash controllers for `io-raw` testing |
| **Miri** | All | Undefined behavior detection in `unsafe` code |
| **Loom** | Tier 2/3 | Exhaustive thread interleaving for `concurrency` code |
| **Host-native** (`io-mem`) | All | Fast unit and integration tests with zero disk I/O |
| **Host-native** (`FailpointIoBackend`) | All | Crash simulation and durability testing |

All environments are available from day 1 and run with `cargo test` or a single `just` command.

### 6.6 Dogfood Applications as Integration Anchors

Three applications consume IonDB as a library and run in CI:

| App | Tier | Validates |
|---|---|---|
| `sensor-log` | Tier 1 | `no_std` compilation, zero-heap operation, binary size, static memory budget |
| `edge-config` | Tier 2 | Transactions, WAL recovery, hash table lookups, heap usage budget |
| `fleet-telemetry` | Tier 3 | Concurrent writes, MVCC isolation, query DSL, `io-file` crash recovery, serde |

If a library change breaks a dogfood app, the PR is blocked. These apps catch API regressions, performance regressions, and feature-flag conflicts that unit tests miss.

## 7. Harness Evolution

The harness is never complete. It evolves through the outer loop:

| Phase | Harness Focus |
|---|---|
| **Phase 0 — Foundation** | Root `CLAUDE.md`, per-crate context files, `no_std` build gate, basic structural tests, `sensor-log` skeleton. |
| **Phase 1 — Storage** | Storage engine property tests, structural invariant checks, `edge-config` skeleton. |
| **Phase 2 — Durability** | Crash simulation harness (`FailpointIoBackend`), WAL correctness tests, Renode flash validation. |
| **Phase 3 — Transactions** | MVCC property tests, Loom integration, `fleet-telemetry` skeleton. |
| **Phase 4 — Buffer Pool** | Write-ahead invariant tests, memory budget enforcement, eviction policy verification. |
| **Phase 5 — Query** | Zero-allocation verification, predicate pushdown correctness, query DSL structural tests. |
| **Phase 6 — Hardening** | Full sweep agent suite, binary size regression tracking, fuzz campaign infrastructure. |
| **Phase 7 — Release** | Documentation staleness sweeps, API surface tests, example compilation tests. |

Each phase adds harness components. The harness compounds — later phases benefit from all earlier investments.

## 8. Success Criteria

The harness is working when:

1. **Agents self-correct.** Most errors are caught and fixed in the inner loop without human intervention.
2. **PRs pass CI on first submission** the majority of the time. Failures are harness gaps, not agent limitations.
3. **No recurring failure patterns.** When a failure type appears twice, it is encoded as a harness component and never appears again.
4. **Code quality is consistent.** Agent-written code is indistinguishable from expert-written code in style, correctness, and architectural alignment.
5. **Cross-tier correctness is automatic.** Code that works on the host also works on Cortex-M0 — verified by CI, not by hope.
6. **The harness compounds.** Each harness improvement benefits every future agent session. Human effort shifts from fixing code to improving the environment.
