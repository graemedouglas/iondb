# IonDB вЂ” Initial Requirements (v0)

## 1. Project Vision

IonDB is an embedded database engine written in Rust, purpose-built for the most resource-constrained environments: microcontrollers, bare-metal systems, RTOS targets, and minimal Linux/embedded-Linux platforms. Every byte of RAM matters. The library must compile and run on targets with as little as a few kilobytes of RAM while scaling gracefully to larger systems when resources are available.

### 1.1 Library-First Packaging

IonDB is delivered **exclusively as a library** (a Rust crate/workspace of crates). There is no standalone binary, daemon, or server process. Users consume IonDB by adding it as a dependency in their own `Cargo.toml` and linking it directly into their application вЂ” whether that application is a `no_std` firmware image, an RTOS task, or a Linux user-space program.

Key implications:

- **No runtime of its own.** IonDB has no `main()`, no event loop, and no background threads unless the user explicitly opts in via feature flags (e.g., `concurrency`).
- **Single address space.** The database lives in the same process as the application. All calls are direct function calls вЂ” no IPC, no sockets, no serialization overhead at the API boundary.
- **Linkable at any layer.** Users may depend on individual workspace crates (e.g., `iondb-storage` alone for a B+ tree without transactions) or on the top-level `iondb` facade crate for the full experience. The multi-crate workspace (В§7.1) is the mechanism that enables this granularity.
- **`no_std` by default.** Because the library is linked into the user's binary, it inherits the user's target constraints. The core crate compiles under `#![no_std]` so that even bare-metal firmware can pull it in.
- **Testable in-process.** Being a library means every component can be unit-tested, fuzz-tested, and property-tested in the host's native test harness (`cargo test`) without standing up an external service. Simulator and emulator environments (В§8.4) extend this to cross-compiled targets.

## 2. Core Design Principles

| Principle | Description |
|---|---|
| **Minimal by default** | The smallest useful build includes only what is strictly needed. Everything else is opt-in via feature flags. |
| **Pay only for what you use** | No hidden allocations, no background threads, no implicit costs. If a feature is not enabled, it contributes zero to binary size and zero to RAM usage. |
| **Pluggable everything** | Storage structures, memory allocators, I/O backends, and serialization are all behind traits so users can swap implementations or provide their own. |
| **Correctness first** | ACID compliance is non-negotiable for the features that are enabled. Every guarantee is backed by tests. |
| **`no_std` compatible** | The core crate must compile under `#![no_std]` with an optional `alloc` gate. `std` features are available but never required. |

## 4. Functional Requirements

### 4.1 Storage Data Structures (Pluggable)

All storage backends implement a common `StorageEngine` trait. Two reference implementations ship with the library:

#### 4.1.1 B+ Tree

- Disk/flash-oriented balanced tree optimized for sequential and range queries.
- Configurable page size (powers of 2, minimum 64 bytes) to match the underlying storage medium's erase block.
- Support for variable-length keys and values with overflow pages.
- Bulk-loading path for efficient initial data ingestion.
- Iterator-based range scans (ascending and descending).

#### 4.1.2 Hash Table

Two hash table variants are provided, each as a separate storage engine implementation:

- **Extendible hashing** (`storage-hash-ext`): Directory-based design with split-on-overflow semantics. Best for workloads with unpredictable key distribution.
- **Linear hashing** (`storage-hash-linear`): Deterministic split order with controlled load factor. Best for steady-growth workloads with predictable insertion rates.

Common properties for both variants:

- Configurable bucket count and load-factor threshold.
- In-place growth without full-table rehash.
- Tombstone-free deletion strategy to avoid scan degradation.

#### 4.1.3 StorageEngine Trait (Common Interface)

```text
trait StorageEngine {
    fn get(key)       -> Option<Value>
    fn put(key, value) -> Result<()>
    fn delete(key)     -> Result<bool>
    fn range(start, end) -> Iterator<(Key, Value)>  // no-op/error for hash
    fn flush()         -> Result<()>
    fn stats()         -> EngineStats
}
```

Users may implement this trait to bring their own storage structures (e.g., LSM tree, skip list, trie).

### 4.2 Memory Management (Pluggable)

#### 4.2.1 Allocator Trait

```text
trait MemoryAllocator {
    fn allocate(layout)   -> Result<*mut u8>
    fn deallocate(ptr, layout)
    fn reallocate(ptr, old_layout, new_layout) -> Result<*mut u8>
    fn available()        -> Option<usize>   // remaining capacity, if known
}
```

#### 4.2.2 Shipped Allocators

| Allocator | Feature Flag | Description |
|---|---|---|
| **Static pool** | `alloc-static` | Fixed-size arena carved from a `static mut` array. Zero heap. Suitable for Tier 1 targets. |
| **Bump allocator** | `alloc-bump` | Fast monotonic allocator with bulk-free. Good for transaction-scoped scratch memory. |
| **System allocator** | `alloc-system` (default with `std`) | Delegates to `std::alloc::GlobalAlloc`. |

Users may implement `MemoryAllocator` to integrate their RTOS heap, custom slab allocator, or external SRAM driver.

### 4.3 I/O Backend (Pluggable)

```text
trait IoBackend {
    fn read(offset, buf)  -> Result<usize>
    fn write(offset, buf) -> Result<usize>
    fn sync()             -> Result<()>
    fn size()             -> Result<u64>
}
```

Shipped implementations:

| Backend | Feature Flag | Description |
|---|---|---|
| **In-memory** | `io-mem` | RAM-backed buffer. Useful for testing and volatile caches. |
| **File** | `io-file` (requires `std`) | Standard filesystem I/O via `std::fs`. |
| **Raw flash** | `io-raw` | Trait adapter for NOR/NAND flash with erase-before-write semantics. Users supply a `FlashDriver` impl. Includes basic wear-leveling enabled by default; can be disabled via configuration when the user's `FlashDriver` handles wear-leveling externally. |

An **async variant** of `IoBackend` is available behind the `async-io` feature flag for use with embedded async runtimes (e.g., Embassy).

### 4.4 ACID Compliance

ACID guarantees are enforced when the `acid` feature flag is enabled.

#### 4.4.1 Atomicity

- Every mutation (single put/delete or a batch within a transaction) is all-or-nothing.
- On crash, uncommitted writes are rolled back during recovery.

#### 4.4.2 Consistency

- The database enforces structural invariants (tree balance, hash-table load bounds) after every committed transaction.
- An optional schema/constraint layer (future, behind `schema` feature flag) validates user-defined rules.

#### 4.4.3 Isolation

- Concurrency control is behind the `concurrency` feature flag (disabled for single-threaded Tier 1 targets).
- When enabled, provide at minimum **Serializable Snapshot Isolation (SSI)** via MVCC.
- **MVCC granularity** is configurable: **row-level** (default) or **page-level**. Row-level avoids false conflicts at the cost of more metadata; page-level is simpler and lower-overhead for coarse-grained workloads.
- When disabled, transactions execute sequentially with no locking overhead.

#### 4.4.4 Durability

- Writes are durable after `commit()` returns, provided the `IoBackend` correctly implements `sync()`.
- Durability can be relaxed via a `durability-level` configuration (e.g., `Immediate`, `Batched`, `None`) to trade safety for throughput on constrained targets.

#### 4.4.5 Write-Ahead Log (WAL)

- All mutations are appended to a sequential log before being applied to storage structures.
- WAL record format: `[CRC32] [TxnID: u64] [Type: u8] [KeyLen: u16] [ValLen: u32] [Key] [Value]`
- Configurable sync policy: `EveryRecord`, `EveryTransaction`, `Periodic(interval)`, `None`.
- Log truncation after checkpoint (background or explicit).
- WAL is required when `acid` is enabled; optional otherwise for crash recovery without full ACID.

#### 4.4.6 Checkpointing

- Flushes dirty pages from the buffer pool to the storage backend.
- Truncates the WAL up to the checkpoint LSN (Log Sequence Number).
- Can be triggered explicitly or on a threshold (WAL size, transaction count).

#### 4.4.7 Recovery

- On startup, replay the WAL from the last checkpoint to restore committed state.
- Detect and discard incomplete transactions.
- Validate structural integrity of storage files via stored checksums.

### 4.5 Buffer Pool / Page Cache

- Fixed-size page cache with configurable page count (can be as small as 2 pages for Tier 1).
- LRU or Clock eviction policy (selectable via feature flag).
- Pin/unpin semantics for pages under active use.
- Dirty-page tracking for efficient checkpointing.
- When `acid` is enabled, integrates with WAL to enforce write-ahead rule (no dirty page flushed before its log record).

### 4.6 Query Interface вЂ” LINQ-Style Embedded DSL

A type-safe, iterator-chain query syntax usable in embedded contexts (no heap-allocated AST, no string parsing at runtime). Two syntax styles are provided вЂ” users choose based on preference:

**Builder pattern** (method chaining):

```rust
let results = db.query()
    .from("sensors")
    .where_(|r| r.field("temp").gt(30))
    .order_by("timestamp", Desc)
    .select(["sensor_id", "temp"])
    .take(10)
    .collect::<Vec<_>>();
```

**Macro syntax** (behind `query-macro` feature flag):

```rust
let results = query! {
    from "sensors"
    where temp > 30
    order_by timestamp desc
    select [sensor_id, temp]
    take 10
};
```

Requirements:

- **Zero-allocation query path** available for Tier 1 (queries operate over iterators, no intermediate collections).
- Builder pattern compiles down to direct storage-engine calls (no interpreter overhead).
- Macro syntax desugars to the builder pattern at compile time.
- Predicate pushdown: filters are evaluated at the storage-engine level when possible.
- Composable: queries can be stored, chained, and reused.
- Optional `query-alloc` feature flag enables richer operations (group-by, join) that require heap allocation.

### 4.7 Serialization

- Key and value types implement a `Codec` trait for encode/decode.
- Ship a minimal built-in binary codec (little-endian, no schema evolution).
- Optional `serde` feature flag for interop with the Serde ecosystem.
- All codecs must support `no_std`.

### 4.8 Transactions

```text
let tx = db.begin()?;
tx.put("key", "value")?;
tx.delete("old_key")?;
tx.commit()?;  // or tx.rollback()
```

- Explicit `begin` / `commit` / `rollback`.
- Nested transactions via savepoints (behind `savepoints` feature flag).
- Read-only transaction mode that skips WAL overhead.
- Auto-rollback on drop if not explicitly committed.

## 6. Non-Functional Requirements

### 6.1 Performance Targets

| Operation | Tier 1 | Tier 2 | Tier 3 |
|---|---|---|---|
| Point read (B+ tree) | < 500 Вµs | < 50 Вµs | < 10 Вµs |
| Point read (hash) | < 200 Вµs | < 20 Вµs | < 5 Вµs |
| Sequential write | < 1 ms | < 100 Вµs | < 20 Вµs |
| WAL append | < 100 Вµs | < 10 Вµs | < 2 Вµs |

*(Targets are order-of-magnitude goals; exact numbers depend on storage medium.)*

### 6.2 Memory Budget

- **Tier 1 baseline**: Core engine operational with в‰¤ 2 KB RAM (excluding user data pages).
- **Static analysis**: `cargo build` with `profile-minimal` must report zero dynamic allocations.
- **Configurable maximums**: Users set hard caps on buffer-pool size, WAL buffer size, and transaction working-set size. The engine stays within these limits or returns an error.

### 6.3 Binary Size

- `profile-minimal` target: < 32 KB `.text` section on `thumbv6m-none-eabi`.
- Dead-code elimination: unused storage engines, allocators, and I/O backends add zero bytes.
- No use of `format!`, `panic!` with formatting, or other string-heavy facilities in `no_std` builds.

### 6.4 Portability

- **Endianness-aware**: All on-disk formats explicitly define byte order (little-endian default).
- **Alignment-safe**: No unaligned reads/writes; all buffer accesses go through safe accessors.
- **No undefined behavior**: `#![forbid(unsafe_code)]` for the core crate, with `unsafe` only in clearly audited leaf modules (`alloc-static`, `io-raw`) behind dedicated feature flags.

### 7.2 Dependency Rules

These rules keep the architecture clean and prevent circular dependencies:

1. **`iondb-core` depends on nothing** (except `core`/`alloc`). It is the leaf of the dependency tree.
2. All other crates depend on `iondb-core` for shared traits and types.
3. **No horizontal dependencies** between implementation crates (e.g., `iondb-storage` must not depend on `iondb-wal`). Cross-cutting integration happens in `iondb-tx` and the facade `iondb` crate.
4. Feature flags propagate upward: enabling `acid` on the facade crate activates the necessary flags in `iondb-wal`, `iondb-tx`, and `iondb-buffer`.
5. `unsafe` code is confined to `iondb-alloc` and `iondb-io` and must be reviewed/audited independently.

### 7.3 Key Architectural Patterns

| Pattern | Where | Why |
|---|---|---|
| **Trait objects vs generics** | Core traits use generics with monomorphization by default; `dyn` dispatch available behind a `dyn-dispatch` feature for binary size reduction. |
| **Typestate pattern** | Transaction lifecycle (`Active` в†’ `Committed` / `RolledBack`) enforced at compile time. |
| **Compile-time configuration** | Page size, max key length, buffer-pool capacity are `const` generics where Rust allows, or associated constants on trait impls. |
| **Error handling** | A single `iondb_core::Error` enum. No `Box<dyn Error>`. Error variants behind feature flags so unused variants are compiled out. |
| **Zero-copy I/O** | Pages are read into caller-provided buffers. No internal `Vec` allocations on the read path. |

### 7.4 Architectural Invariants (Enforced by CI)

- `cargo build --no-default-features --target thumbv6m-none-eabi` must succeed for `iondb-core`, `iondb-alloc`, `iondb-storage`, `iondb-wal`, `iondb-buffer`.
- `cargo clippy` with `-D warnings` on all feature combinations (tested via a feature-matrix CI job).
- No `std` import reachable when `std` feature is disabled (enforced by the `thumbv6m` build).
- Dependency audit: no transitive dependency pulls in `std` unless the `std` feature is active.

### 7.5 Code Quality Requirements (Enforced by CI)

All of the following must pass in CI before any PR can be merged. **Zero tolerance for violations.**

- **100% unit test coverage**: Every crate must maintain 100% line coverage in unit tests. Measured via `cargo-tarpaulin` or `llvm-cov`. Coverage reports are generated per PR and any drop below 100% blocks the merge.
- **Auto-formatting**: All code must pass `cargo fmt --all -- --check` with no changes required. The project uses the default `rustfmt` configuration. Formatting is enforced automatically вЂ” no manual style review needed.
- **Zero warnings**: `cargo build` and `cargo test` must produce **zero compiler warnings** across all feature combinations. CI builds use `RUSTFLAGS="-D warnings"` to promote warnings to errors.
- **Zero Clippy warnings**: `cargo clippy --all-targets --all-features -- -D warnings` must pass cleanly. No `#[allow]` annotations to suppress Clippy lints without a justifying comment approved in code review.
- **Zero doc warnings**: `RUSTDOCFLAGS="-D warnings" cargo doc --no-deps` must pass with no warnings. All public APIs must have documentation.

## 8. Testing Strategy

### 8.1 Test Levels

| Level | Location | Scope | Runner |
|---|---|---|---|
| **Unit tests** | `#[cfg(test)]` in each module | Single function or struct | `cargo test` |
| **Integration tests** | `tests/integration/` | Cross-crate workflows (e.g., insert в†’ WAL в†’ crash в†’ recovery в†’ read) | `cargo test` |
| **Property-based tests** | `tests/property/` | Invariant verification over random inputs (using `proptest` or `quickcheck`) | `cargo test --features test-proptest` |
| **Fuzz tests** | `tests/fuzz/` | Crash/corruption discovery via `cargo-fuzz` / `libfuzzer` | `cargo fuzz run` |
| **Embedded simulation** | CI via QEMU | Verify `no_std` builds run correctly on emulated Cortex-M | Custom CI step |
| **Benchmarks** | `benches/` | Performance regression tracking (using `criterion`) | `cargo bench` |

### 8.2 What Must Be Tested

#### Storage Engines (per engine)
- Insert, read, update, delete for all key/value sizes (empty, 1 byte, max length, overflow).
- Range scan correctness (ascending, descending, partial range, empty range).
- Behavior at capacity limits (disk full, max keys).
- Structural invariants after every mutation (tree balance, hash load factor).
- Concurrent access (when `concurrency` enabled): no lost updates, no phantom reads.

#### WAL
- Append correctness: records round-trip through encode/decode.
- CRC validation: corrupted records are detected and skipped during recovery.
- Truncation: WAL shrinks correctly after checkpoint.
- Crash simulation: kill process mid-write, verify recovery replays only complete records.

#### Transactions
- Commit persists all writes.
- Rollback discards all writes.
- Auto-rollback on drop.
- Savepoint: partial rollback to savepoint, then commit remainder.
- Isolation: concurrent transactions see correct snapshots (MVCC property tests).

#### Buffer Pool
- Eviction policy correctness (LRU/Clock ordering).
- Dirty-page tracking: dirty pages are written before eviction.
- Write-ahead invariant: no dirty page is flushed before its WAL record is synced.
- Pin count: pinned pages are never evicted.

#### Allocators
- Static pool: allocate up to capacity, then fail gracefully.
- Bump: reset frees all memory; allocations after reset succeed.
- Alignment: every returned pointer satisfies the requested alignment.

#### Query DSL
- Predicate evaluation correctness for all comparison operators.
- Order-by produces sorted output.
- Take/skip/limit produce correct subsets.
- Zero-allocation path: queries over iterators allocate nothing (verified via `#[global_allocator]` counting allocator in tests).

#### Recovery
- Clean shutdown: no recovery needed, startup is fast.
- Crash after WAL write but before page flush: data is recovered.
- Crash during checkpoint: partial checkpoint is detected, WAL replay completes.
- Corrupted storage file: detected via checksums, reported as error.

### 8.3 Test Infrastructure

- **Crash simulation harness**: A `FailpointIoBackend` wrapper that injects I/O failures at configurable points (before write, after write, during sync) to exercise every recovery path.
- **Counting allocator**: A `#[global_allocator]` wrapper that counts allocations, used in tests to verify zero-allocation guarantees.
- **Deterministic random seeds**: All property tests and fuzz targets use reproducible seeds for CI bisection.
- **Feature-matrix CI**: A CI job that builds and tests every meaningful combination of feature flags (automated via a script, not manually maintained).
- **Coverage target**: **100% line coverage** on unit tests across all crates. Coverage reports published per PR. No PR may be merged that decreases coverage below 100%.

### 8.4 End-to-End Simulator Environments (Day 1)

Cross-target testing must be available **from the first commit**, not deferred to a hardening phase. The following simulator and emulation environments are integrated into CI and available locally from day 1.

#### 8.4.1 QEMU System Emulation (Tier 1 & 2)

| Target | QEMU Machine | Notes |
|---|---|---|
| `thumbv6m-none-eabi` (Cortex-M0) | `lm3s6965evb` (TI Stellaris) | Primary Tier 1 validation target. Runs bare-metal test binaries via `qemu-system-arm`. |
| `thumbv7em-none-eabihf` (Cortex-M4F) | `netduinoplus2` (STM32F4) | Tier 2 validation with hardware FPU. |
| `riscv32imac-unknown-none-elf` | `virt` (RISC-V) | Validates RISC-V `no_std` builds. |

**Integration approach:**

- Use [`defmt-test`](https://github.com/knurling-rs/defmt) as the embedded test harness вЂ” it supports `#[test]`-like macros for `no_std` targets and emits structured log output via `defmt`.
- Use [`probe-run`](https://github.com/knurling-rs/probe-run) or [`qemu-run`](https://github.com/nickelc/qemu-run) as the Cargo runner so that `cargo test --target thumbv6m-none-eabi` transparently boots QEMU, loads the test binary, captures output, and reports pass/fail.
- Configure `.cargo/config.toml` per target with the appropriate runner, so developers run `cargo test --target <triple>` and QEMU launches automatically.

#### 8.4.2 Renode (Multi-Peripheral Simulation)

[Renode](https://renode.io/) provides cycle-accurate simulation of full SoCs including flash controllers, UART, SPI buses, and DMA вЂ” enabling end-to-end tests that exercise the `io-raw` flash backend against a simulated NOR/NAND device rather than a mock.

| Use Case | How |
|---|---|
| Flash I/O validation | Define a Renode platform script (`.resc`) with an emulated SPI NOR flash chip. The test firmware writes through `IoBackend::write()` and reads back via `IoBackend::read()`, verifying round-trip correctness including erase-before-write semantics. |
| Wear-leveling verification | The Renode flash model tracks per-sector erase counts. After a test run, assert that erase counts are balanced within tolerance. |
| Multi-device scenarios | Simulate two MCUs sharing a flash chip (via SPI) to test concurrent access corner cases. |

**CI integration:** Renode runs headless via `renode --disable-xwt --execute "start; ... ; quit"`. A thin shell wrapper translates test results into CI-compatible exit codes.

#### 8.4.3 Host-Native Simulation (Tier 3 & General)

Not all tests require cross-compilation. The following host-native environments cover Tier 3 and general correctness:

| Environment | Purpose |
|---|---|
| **In-memory I/O backend** (`io-mem`) | Default for all unit and integration tests. Zero disk I/O, deterministic, fast. Available from day 1 as a first-class `IoBackend` implementation. |
| **Temporary-directory file backend** | For `io-file` tests: create a `tempdir`, run the test, assert file contents, clean up. Uses Rust's `tempfile` crate. |
| **`FailpointIoBackend`** (В§8.3) | Wraps any `IoBackend` and injects faults (I/O errors, partial writes, sync failures) at configurable points. Essential for crash-recovery and durability testing without requiring actual power cuts. |
| **Miri** | Run `cargo +nightly miri test` on allocator and `unsafe` code to detect undefined behavior, use-after-free, and alignment violations. Integrated into CI from day 1. |
| **Loom** | For `concurrency`-flagged code: use [`loom`](https://github.com/tokio-rs/loom) to exhaustively explore thread interleavings and catch data races, deadlocks, and ordering bugs in MVCC and transaction code. |

#### 8.4.4 CI Configuration

All simulator environments are defined in CI from the initial repository setup:

```text
ci-matrix:
  host-native:
    - cargo test                                    # all host-native unit + integration tests
    - cargo +nightly miri test -p iondb-alloc       # UB detection in allocators
    - cargo test --features concurrency -- loom      # thread interleaving tests

  qemu-arm:
    - cargo test --target thumbv6m-none-eabi         # Tier 1 via QEMU
    - cargo test --target thumbv7em-none-eabihf      # Tier 2 via QEMU
```

The goal is **zero manual setup**: a developer clones the repo, installs the documented toolchain (via `rust-toolchain.toml` and a `Makefile`/`justfile` target for QEMU/Renode), and every test вЂ” host-native through cross-compiled вЂ” runs with a single command.

## 10. Decisions (Resolved)

1. **Hash table variant**: Ship both extendible hashing and linear hashing as separate storage engine implementations. Each is optimized for different workloads вЂ” users select the variant that best fits their access patterns.
2. **MVCC granularity**: Support both page-level and row-level versioning. Row-level is the **default** (avoids false conflicts); page-level is available as an opt-in for scenarios where simpler metadata and lower overhead are preferred. Selectable via configuration.
3. **Query DSL syntax**: Ship both the macro-based (`query!{}`) syntax and the pure builder pattern. Users choose based on preference вЂ” macros for ergonomics, builder for clearer error messages and IDE support.
4. **Flash wear leveling**: Yes вЂ” include basic wear-leveling in `io-raw`, enabled by default. Users can disable it via configuration when they provide their own wear-leveling in their `FlashDriver` implementation.
5. **Async I/O**: Yes вЂ” the `IoBackend` trait will have an async variant for use with embedded async runtimes (e.g., Embassy), gated behind an `async-io` feature flag.

## 11. Dogfooding: User Applications

IonDB is developed **alongside** one or more user-facing applications that consume it as a library. These applications serve as continuous, real-world validation of the database API, performance characteristics, and ergonomics. Development of dogfood applications begins in **Phase 0** and evolves in lockstep with the library.

### 11.1 Purpose

- **API ergonomics feedback.** Using IonDB as a real consumer reveals awkward type signatures, missing convenience methods, confusing error variants, and documentation gaps far earlier than unit tests alone.
- **Integration-level coverage.** Dogfood apps exercise cross-crate interactions (storage + WAL + transactions + queries) in realistic sequences that synthetic integration tests may not cover.
- **Performance profiling under realistic workloads.** Benchmarks measure micro-operations in isolation; dogfood apps generate representative mixed read/write/scan traffic for profiling.
- **Tier validation.** Each app targets a specific hardware tier, ensuring IonDB meets its constraints (memory budget, binary size, `no_std` compatibility) in practice, not just in theory.

### 11.4 CI Integration

Dogfood applications are **first-class CI citizens**:

- **Build check**: Every PR must build all apps for their respective targets (cross-compile for Tier 1/2, host-native for Tier 3).
- **Functional tests**: Each app includes scenario-based tests (e.g., `sensor-log`: insert 1000 readings в†’ query last 10 в†’ assert correctness). These run on the appropriate simulator (QEMU, Renode, or host).
- **Resource-budget tests**: Tier 1/2 apps assert binary size and peak RAM usage via CI scripts that parse linker map files and memory reports.
- **Regression detection**: If a library change breaks a dogfood app, the PR is blocked. This catches API regressions, performance regressions, and feature-flag conflicts before they reach external users.
